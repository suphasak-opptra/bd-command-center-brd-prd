2.4 Notion Database Schema (Unified Single-Table)

Architecture Decision: This system uses a single unified database instead of the traditional 4-database CRM architecture (Deals + Brands + Activity Log + Contacts). Activity logs and documents are stored as page content within each row. Phase 2 (Weeks 3-4) introduces linked databases once the team is comfortable with the core workflow.

Activity Log Architecture

What replaced the Activity Log Database?

Notion's built-in page content. Each brand-geo row has its own dedicated page that serves as an activity log. Meeting notes, call summaries, commitments, documents, and activity history are stored within that page body. This approach:

  • Keeps all information about a deal in one place (the deal page itself)
  • Reduces database complexity and admin overhead
  • Makes it easy for the team to see the full history of a deal by opening one page
  • Supports phase 2 migration to a dedicated linked Activity Log database when needed

Contacts Architecture

What replaced the Contacts Database?

Brand-side contacts are stored directly as fields on each brand-geo row (Brand Owner, Brand Contact). Internal team contacts (BD Owner, Channel Lead, Brand Vertical Lead) are Person fields that link to your Notion workspace members. This approach:

  • Keeps deal and contact information together on the same row
  • Eliminates the need for a separate Contacts database for the initial workflow
  • Makes it easy to see who the key contacts are without navigating to a separate table
  • Supports phase 2 migration to a dedicated linked Contacts database for multi-deal relationships

2.13 Event Triggers Across Pipeline Stages

Each pipeline stage transition is driven by a specific, observable event. These triggers were defined with the BD team based on real user behavior and mapped to available automation capabilities.

Transition Trigger Event Detection Method Automation Secondary KPI
Wishlist → Lead First brand response (email reply, LinkedIn, WhatsApp) Gmail scan for brand-domain replies Semi-auto Response rate, time-to-first-response
Lead → MQL First meeting scheduled (Google Calendar event with external attendees) GCal scan — external attendee detection Auto-detectable ✅ Meeting conversion rate, days lead-to-meeting
MQL → SQL Deck preparation email sent to Kriti (design team) Gmail scan for "deck" + Kriti's email Semi-auto Pitch prep cycle time
SQL → Opportunity CEO approval given (verbal/Slack/email confirmation) Slack message detection in BD channel Semi-auto CEO approval cycle time
Opportunity → Signed Signed contract PDF uploaded (Google Drive or email attachment) Drive scan for signed PDF in brand folder Semi-auto Negotiation duration, contract turnaround
Key insight from user research: BD team members are already creating Google Calendar events for brand meetings. This makes Lead → MQL the most reliably auto-detectable transition — the calendar event IS the signal, no behavior change required from the team.

2.14 GCal Brand Meeting Detector (Live)

The first production automation: a scheduled task that scans Google Calendar for brand meetings and posts matches to Slack for human approval before updating Notion.

How It Works

Step Action Tool

2.15 Automation Architecture & HITL Workflow

All automations follow the same 4-step human-in-the-loop pattern. This ensures no pipeline data is modified without team oversight.

Automation Readiness by Stage Transition

Category Transitions Status
Auto-detectable Lead → MQL (GCal external attendee detection) Live — scheduled task deployed
Semi-auto (planned) Wishlist → Lead (Gmail reply scan), MQL → SQL (deck email to Kriti), SQL → Opportunity (CEO Slack approval), Opportunity → Signed (signed PDF in Drive) Event triggers defined, automation not yet built

HITL Guardrails

  • No silent writes: Every detected event posts to Slack before any Notion update
  • Approval required: Human must react ✅ in Slack to authorize pipeline changes
  • Rejection path: ❌ reaction logs the rejection, no pipeline change made
  • Timeout: No reaction within 24 hours = no action (logged as "no response")
  • Audit trail: All detections, approvals, and rejections logged in Slack history
  • Graduation path: After 2+ weeks of validated accuracy, proven automations can graduate to auto-execute with post-hoc notification (Phase 2)

2.16 Phase 2 Enhancements (Weeks 3-4)

Once the team is comfortable with the unified single-table workflow, introduce these linked database enhancements to add advanced capabilities:

1. Dedicated Activity Log Database

As the core system stabilizes, introduce a linked Activity Log database for formal audit trails. Benefits:

  • Timestamped stage change history with clear before/after states
  • Actor attribution for every update (who changed what, when)
  • SLA breach event logging for compliance reporting
  • Compliance-ready audit records that survive deal page deletions

Implementation: Link each BD Pipeline row to an Activity Log; Claude updates both on every stage change or significant event.

2. Contacts Database

As the brand network grows beyond 200+ contacts, split contacts into a dedicated relational database. Benefits:

  • One contact linked to multiple brand-geo deals
  • Relationship status tracking (New → Warm → Champion)
  • Contact-level activity history and communication preferences
  • Ability to see all deals a contact is involved in

Implementation: Create a Contacts DB linked bidirectionally to BD Pipeline. Move Brand Owner and Brand Contact fields to relation references.

3. Documents & Knowledge Base Database

Create a linked Documents database for brand analyses, pitch decks, scouting reports, and proposals. Benefits:

  • Prevents duplication when the same brand is pursued across multiple geos
  • Creates a searchable knowledge base of all brand research
  • One document linked to multiple deals
  • Audit trail of document edits and versions

Implementation: Create a Documents DB with brand-level grouping; link to both Contacts and BD Pipeline rows.

4. Advanced Claude Automations (HITL Graduation)

Graduate proven HITL automations from "notify → approve → update" to "auto-execute → notify" after 2+ weeks of validated accuracy. The GCal brand meeting detector (section 2.14) is the first candidate for graduation.

  • Auto-stage advancement: Graduate Lead→MQL from HITL to auto-execute once approval rate exceeds 95% for 2 consecutive weeks
  • Auto-revenue calculation: Auto-update revenue estimates when brand data changes
  • Auto-rejection: For clear red flags, auto-reject deals with reason code (with manual override option)
  • Auto-outreach: Generate draft emails/templates for outreach based on deal profile and stage
  • Remaining semi-auto triggers: Build Gmail reply scan (Wishlist→Lead), deck email detection (MQL→SQL), CEO approval capture (SQL→Opp), signed PDF detection (Opp→Signed)

Safety gate: Only enable after 2+ weeks of validated accuracy in HITL mode and explicit team approval. Each transition graduates independently.

5. Dashboard Views & Reporting

Build advanced Notion views and reporting capabilities:

  • Pipeline Board: Kanban view grouped by stage with drag-to-advance
  • Revenue Forecast: Weighted pipeline by quarter with scenario modeling
  • SLA Heatmap: Color-coded compliance matrix (red = breached, yellow = warning, green = on track)
  • Team Workload: Deals per owner, average cycle time, SLA compliance per person
  • Automated digests: Generate weekly, monthly, and quarterly summaries with trend analysis

6. Slack & Gmail Deep Integration

Enhance real-time deal updates and communication:

  • Email thread linking: Auto-link email threads to brand pages, detect mentions, show deal context in email
  • Slack commitments: Capture Slack commitments as deal notes and set reminders
  • Pipeline broadcasts: Auto-post stage transitions to #bd-pipeline channel with stakeholder pings
  • Daily standup: Generate daily standups from overnight changes and upcoming deadlines
